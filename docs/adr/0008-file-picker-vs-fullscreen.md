# ADR-0008: ファイル選択に File System Access API を使わない

日付: 2026-06-13
ステータス: 採用（フルスクリーン問題の根本対策は ADR-0009。本 ADR は補助的な変更）

## コンテキスト

出力ウィンドウをフルスクリーンにした状態で、コントローラの「ファイルを追加」等を押すと、出力ウィンドウのフルスクリーンが解除され、再生がフリーズする不具合が報告された。

当初、原因は `window.showOpenFilePicker` / `showDirectoryPicker`（File System Access API）がフルスクリーンを解除する仕様にあると考え、ファイル選択を `<input type="file">` に切り替えた。しかし**実機では `<input>` でも再発した**。真因はファイルダイアログ（OS ネイティブ）が前面に出ること自体であり、API 種別とは無関係だった（根本対策は ADR-0009）。

それでも `<input>` への変更自体は次の理由で維持する価値がある:

- `showOpenFilePicker` も呼び出し時にフルスクリーンを解除するため、`<input>` の方が少なくとも悪化しない
- File System Access API への依存が減る

ADR-0004 では、ライブラリ永続化のためにこの API でファイルハンドルを取得していた。

## 決定

「ファイルを開く」「ファイルを追加」「フォルダを追加」はすべて `<input type="file">` 経由にする。

ライブラリの永続化（`FileSystemFileHandle` の IndexedDB 保存）は、**ドラッグ＆ドロップ経由でのみ**維持する。ドロップ時の `DataTransferItem.getAsFileSystemHandle()` は使い続ける。

## 理由

- ボタンの裏が `<input>` でも `showOpenFilePicker` でも、OS のファイルダイアログは開く。どちらでもフルスクリーン問題は起きるため、選択は他の観点（依存の少なさ）で決めてよい
- 永続化を完全に失うわけではなく、ドラッグ＆ドロップに導線を移すことで両立できる。VJ ではフォルダごとドロップする方が運用上も自然

## 結果

- ボタン経由で追加したエントリは常にセッション内のみ有効（永続化されない）。永続化したい場合はドラッグ＆ドロップで追加する
- File System Access API 非対応ブラウザでは、ドロップでも永続化されない（従来どおり）
- フォルダ選択は `<input webkitdirectory>` を使う。サブフォルダも含めて再帰的に列挙される点が `showDirectoryPicker`（直下のみ）と異なるが、mp4 のみ採用するため実害は小さい

## 教訓

GUI・OS ダイアログ・ウィンドウ管理が絡む不具合を、ヘッドレスブラウザの自動テストだけで「解決した」と判断したのが誤りだった。ヘッドレスでは OS ダイアログが出ないため、本質を再現できていなかった。この種の不具合は実機での観測（ADR-0009 に記録）を必須とする。
