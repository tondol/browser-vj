# ADR-0010: ライブラリの永続化を廃止し、メモリ保持に統一する

日付: 2026-06-13
ステータス: 採用（ADR-0004 を置き換え）

## コンテキスト

ADR-0004 ではライブラリ永続化のため `FileSystemFileHandle` を IndexedDB に保存していた。しかし ADR-0008 で File System Access API（`showOpenFilePicker`）をフルスクリーン問題のため使わなくした結果、永続化はドラッグ＆ドロップ経由でしか効かなくなり、導線が分かりにくくなっていた。

加えて、永続化のためにコードが複雑化していた:

- IndexedDB とメモリの二重管理（`Library` クラス）
- ハンドルの読み取り権限プロンプト（`queryPermission` / `requestPermission`）
- ドロップ時に `getAsFileSystemHandle()` を同期取得する処理。この複雑さがフォルダのドラッグ＆ドロップが動かない原因にもなっていた（フォルダは `getAsFile()` で展開できない）

VJ の実運用では、ライブラリは毎回その場のフォルダから組み立てる使い方が中心で、永続化の必要性は高くないと判断した。

## 決定

ライブラリの永続化を廃止し、**セッション内のメモリ保持のみ**にする。`Library` は `File[]` を持つだけの単純なクラスにする。

フォルダのドラッグ＆ドロップは `DataTransferItem.webkitGetAsEntry()` で `FileSystemEntry` を取得し、ディレクトリを再帰的にたどって mp4 を集める方式にする（`readEntries` は全件返らないことがあるため空になるまで繰り返す）。エントリが取れない環境では `dataTransfer.files` にフォールバックする。

## 理由

- 永続化を捨てることで、IndexedDB・権限プロンプト・ハンドル取得という複雑さが一掃され、`library.ts` が大幅に単純化される（バンドルも縮小）
- ハンドルを同期取得する制約が消えるため、フォルダ展開（`webkitGetAsEntry`）を素直に実装できる
- 永続化が無くても、フォルダごとドロップすればライブラリは数秒で再構築できる

## 結果

- ブラウザを閉じる／リロードするとライブラリは空になる。UI にその旨を常時注記する
- 動画ハンドルではなく `File` を保持するため、ライブラリ表示中もファイルを掴み続ける（メモリ上は Blob 参照のみで、動画データの複製はしない）
- `src/types/fs-access.d.ts`（File System Access API の型補完）は不要になり削除した
