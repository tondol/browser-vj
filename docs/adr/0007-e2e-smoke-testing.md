# ADR-0007: テスト方針（実ブラウザ E2E スモークテスト）

日付: 2026-06-12
ステータス: 採用

## コンテキスト

本アプリの中核的な振る舞いは、DOM・`<video>` 要素・BroadcastChannel・ブラウザの省電力挙動といった「実ブラウザでしか再現しない領域」に集中している（マルチウィンドウ同期、フェーダー合成、ループ制御、バックグラウンド凍結耐性など）。これらを継続的に壊さず開発するための検証手段が必要。

論点:

1. テストの粒度（ユニット / E2E）
2. テスト実行基盤（フレームワーク導入 / 自前）
3. ランタイム依存ゼロ方針（ADR-0001）との整合

## 決定

- **実ブラウザを操作する E2E スモークテストを主軸にする。** ヘッドレス Chrome を puppeteer で駆動し、実アプリに対してロード・再生・同期・ホットキーなどを行って結果を検証する
- **テストフレームワークは導入せず、薄い自前ランナーで済ませる。** `test/run.mjs` が dev サーバの起動・待機・実行・停止を行い、各テストは `node:assert` 相当の最小チェッカで PASS/FAIL を出す
- **`puppeteer-core` を devDependency として追加する。** ランタイム依存はゼロのまま維持する
- **テスト用動画は ffmpeg で生成し、無ければ SKIP する。** Chrome の実行パスは `CHROME_PATH` で上書き可能にする

## 理由

- VJ ツールの価値はマルチウィンドウの映像同期や手動リップシンクにあり、これらは関数単体テストでは品質を保証できない。実ブラウザでの観測が必須
- Vitest や Playwright Test を入れれば機能は揃うが、本アプリの規模（数ファイル）に対して依存と設定が過剰。dev サーバ起動と puppeteer 駆動さえあれば要件を満たせる
- `puppeteer-core`（ブラウザ同梱なしの軽量版）はシステムの Chrome を使うため、巨大な Chromium ダウンロードを伴わない。テスト専用の依存なので配布物・実行時には影響しない

## 結果

- テストの実行には Chrome（既定で macOS の Google Chrome）と、動画生成のための ffmpeg が必要。CI で回す場合はこれらの用意が前提になる（TODO: Linux CI 上での Chrome/ffmpeg セットアップは未整備）
- フレームワーク非依存ゆえに、並列実行・リトライ・リッチなレポートは自前で用意しない限り得られない。現状の規模では不要と判断
