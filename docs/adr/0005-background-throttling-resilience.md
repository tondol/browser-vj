# ADR-0005: バックグラウンド省電力による強制停止への対策

日付: 2026-06-12（同日改訂）
ステータス: 廃止（2026-06-13、ADR-0009 で前提が消えたため。下記「廃止の経緯」参照）

## コンテキスト

出力ウィンドウをフルスクリーン化（macOSでは別Spaceへ移動）すると、コントローラウィンドウが完全に隠れ、Chromeが省電力のためミュート動画の再生を止めることがある。ADR-0003の設計ではコントローラの`video`要素がマスタークロックなので、これがそのまま出力ウィンドウに波及し、観客に見えている映像まで止まってしまう。

さらにChromeの停止のしかたは一通りではない:

- `pause`イベントを発火して止める
- **`pause`イベントを発火せず、`currentTime`の進行だけを凍結する**（`paused`は`false`のまま）

候補:

1. `pause`イベントを検知して意図しない停止だけ握りつぶし、停止中は実時間外挿で同期する
2. コントローラが`video`要素に依存しない実時間ベースのマスタークロックを持ち、`video`はクロックに追従するプレビューに格下げする
3. マスタークロックを出力ウィンドウへ移す
4. Wake Lock API等でスロットリング自体を抑止する

## 決定

**案2（実時間ベースのマスタークロック）** を採用する。

- 各デッキは `playing` と アンカー（メディア時刻＋実時刻のペア）だけを真実として持ち、現在の再生位置は常に `anchor + 経過実時間（ループ分は剰余）` で導出する
- アンカーはユーザー操作（ロード・再生/停止・シーク・ナッジ）のときだけ更新する
- `sync`・`hello` で配信する再生状態はすべてこのクロックから導出する（`video.currentTime`は使わない）
- コントローラの`video`はプレビューとして毎フレーム クロックと比較し、ずれたら追従させる。意図しない`pause`イベントには再開を試みるだけで、クロックには影響させない

## 理由

- 当初は案1を実装したが、`pause`イベントなしで`currentTime`だけ凍結するケースを検知できず、「凍結した時刻＋playing=true」を配信して出力側が約1秒区間（隠れタブのタイマー間隔）をループする不具合が出た
- 案2はブラウザがどんな方法で再生を止めても影響を受けない。停止の検知自体が不要になるため、ブラウザ実装の変化にも頑健
- 案3はADR-0003の「出力ウィンドウはいつ閉じて開き直してもよい」性質を壊す。案4は発動条件がブラウザ依存で保証がない

## 結果

- 真実の状態が`video`要素から分離されたことで、コントローラのUI表示（再生ボタン等）はユーザー操作時に直接更新する必要がある
- コントローラが隠れている間の同期配信はタイマースロットリングで疎になる（最大1回/分）が、クロックは実時間導出なのでズレは蓄積しない
- 動画のメディアクロックと実時間のごく僅かな乖離は、プレビュー側は0.3s閾値の追従で、ミラー側はADR-0003のドリフト補正で吸収する

## 廃止の経緯（2026-06-13）

この対策が必要だった前提は「コントローラウィンドウが `visibilityState=hidden` になり、ブラウザが `video` を凍結する」こと。そして `hidden` になるのは出力ウィンドウのフルスクリーン化（別 Space 送り）が引き金だった。

ADR-0009 でフルスクリーン API をやめ、ウィンドウ最大化に変更したことで、この引き金が消えた。実機で検証したところ:

- コントローラの最小化・別ウィンドウで被覆・別タブ切り替え、いずれでも `visibilityState` は `visible` のままで、`video.currentTime` は凍結しなかった
- 実時間クロックを無効化（`video.currentTime` 直接参照）した版でも、上記操作で出力映像は止まらなかった

つまり現在の運用では実時間クロックは何も守っていないことが確認できた。マルチウィンドウ同期は `video.currentTime` を直接配信する素朴な実装で十分なため、クロック（アンカー＋実時間導出）を撤去してコードを単純化した。ループオフ時の終端停止は `video` の `ended` イベントで扱う。

※ もし将来 `hidden` で凍結する経路が再び現れた場合（プロジェクタ運用でのディスプレイスリープ等）は、本 ADR の手法を再導入する余地がある。

※（2026-06-20 追記）スマホ操作で全画面のままコントローラ窓を長時間裏に置く運用により、この凍結経路が実際に再来した。ADR-0011 では実時間マスタークロックの再導入ではなく、デッキ動画に音声を載せて `video-only` の省電力停止自体を回避する方式を採った。本 ADR の手法（案 2）は、それでも凍結する経路が現れた場合の次の手段として残る。
