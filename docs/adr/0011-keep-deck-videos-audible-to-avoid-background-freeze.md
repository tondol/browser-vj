# ADR-0011: デッキ動画をミュートせず音声を載せ、全画面時の省電力停止を防ぐ

日付: 2026-06-20
ステータス: 採用（ADR-0005 が想定した「凍結経路の再来」への対処）

## コンテキスト

スマホをリモコンにして PC で全画面出力する運用（WebSocket ブリッジ経由でコントローラを遠隔操作する）を追加した。この運用では、出力ウィンドウを最大化して前面に置き、**コントローラ（操作 UI）ウィンドウを長時間裏に置いたまま**にする。

この状態で、出力映像が約 0.5 秒周期で巻き戻る不具合が出た。実機で観測した事実:

- コントローラウィンドウのコンソールに `AbortError: The play() request was interrupted because video-only background media was paused to save power.`
- 出力ウィンドウ側には `sync` が届いていない（`applySync` が呼ばれない）
- 全画面のときだけ発生し、コントローラ窓をアクティブにすると解消する

判明したこと:

- Chrome は**音声トラックを持たない（`video-only`）動画**を、ウィンドウが裏に回ると省電力のため再生停止する
- デッキ動画（`video-a` / `video-b`）は `muted` だったため、この停止対象になっていた
- 停止すると `video.currentTime` が進まないまま、500ms ごとの `sync` ループは「凍結した時刻＋`playing=true`」を送り続ける。前面で再生が進む出力ウィンドウは、止まった `time` との差が `DRIFT_HARD_SEEK_SEC`(0.5s) を超えるたびにハードシークで巻き戻り、0.5 秒周期のループになる

これは ADR-0005 が解決し、ADR-0009 で前提が消えたとして撤去した不具合と同一の機序である。ADR-0005 は廃止時に「将来 `hidden` で凍結する経路が再び現れた場合は本手法を再導入する余地がある」と注記していた。今回のスマホ操作運用（全画面で長時間裏に置く）が、まさにその凍結経路を再来させた。

## 決定

デッキ動画（`video-a` / `video-b`）から `muted` を外し、**音声を載せて `video-only` 判定を回避する**。これによりウィンドウが裏に回っても省電力停止の対象にならず、コントローラの再生・`sync` 配信が止まらない。

スピーカーは OS 側でミュートする運用を前提とし、アプリ内では消音しない。プレビュー（`preview-a` / `preview-b`）とサムネ生成用の一時 `video` は同一ウィンドウ内で裏に回らず、音も不要なため `muted` のままとする。

## 理由

- ADR-0009 の「ウィンドウ最大化方式」は維持したい（Fullscreen API は macOS で専用 Space を作り、ファイルダイアログで裏に隠れて描画が止まる別問題を持つ。これは音声では直らない）。最大化方式の弱点である「裏窓の省電力停止」だけを音声で塞ぐのが、既存方式への影響が最も小さい
- ADR-0005 の実時間マスタークロック（案 2）を再導入する手もあるが、撤去でせっかく単純化したコードを再び複雑にする。`muted` を外すだけの一行で凍結経路自体を断てるなら、そちらが軽い
- 「手元で動けばよい」用途で、出力スピーカーは OS でミュートできるため、音を鳴らしっぱなしにする副作用は実用上問題にならない

## 結果

- デッキ動画の音声が再生される。出力スピーカーは OS 側でミュートする前提（アプリは消音しない）。A/B 両デッキが鳴るため音は重なるが、OS ミュート運用のため実害はない
- 音ありの再生はブラウザの autoplay ポリシーの対象になる。動画ロードは必ずユーザー操作（ファイル選択・ライブラリボタン）起点なので最初の再生は通る。一度ユーザー操作で再生が始まればページは音声再生の許可を得るため、以降スマホ経由（ユーザージェスチャー外）のロードでも再生できる
- 省電力停止の機序は Chrome のバージョンで挙動差がありうる。もし `video-only` 回避でも凍結する経路が現れた場合は、ADR-0005 の実時間マスタークロック（案 2）の再導入が次の手段になる
